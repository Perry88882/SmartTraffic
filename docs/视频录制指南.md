## 二、逐段讲解脚本

### 2.1 开场（0:00-0:40）

**画面：** 展示仪表盘首页全貌。建议先录一张仪表盘截图作为封面，然后切到实时画面。

**要讲的内容：**

> "各位老师好，我做的项目是 SmartTraffic——基于机器学习的智能网络流量安全分析系统。"
>
> "这个系统解决了传统流量分析工具的三大痛点。第一，Wireshark 等工具需要网络专家逐个包手动分析，学习成本非常高。第二，现在互联网流量中 HTTPS 加密占比已经超过 95%，传统的深度包检测技术无法穿透 TLS 加密层。第三，市面上缺少一个集成了实时分析、安全评估、历史回溯的一体化平台。"
>
> "我们的系统实现了从物理层抓包到应用层 AI 分类的全链路自动化分析，并且加入了五层模型可视化、安全风险评估和 GeoIP 物理地址识别等扩展功能。"

**操作：**
- 显示仪表盘首页
- 鼠标缓慢划过四个统计卡片、两个图表、记录表格

### 2.2 系统架构（0:40-1:30）

**画面：** 打开 `docs/实验报告.md` 或 `SmartTraffic项目设计报告.md`，滚动到架构图部分。或者直接在浏览器中展示，用画图工具标注。

**要讲的内容：**

> "系统采用管道化架构，从上到下分为三层。"
>
> "最上层是浏览器前端，使用 React 18 加 Vite 构建，通过 HTTP REST 和 WebSocket 两种协议与后端通信。仪表盘包含七个组件——控制栏、统计卡片、饼图、面积图、记录表格、数据包详情面板，以及历史记录弹窗。"
>
> "中间层是 Flask 后端服务器，这也是系统的核心。它包含三个子系统：REST API 负责处理前端的启停控制和历史查询请求；WebSocket 负责实时推送分类结果和统计数据；数据管道是整个系统的大脑，包含六个处理阶段——抓包管理、协议解析、流重组、特征提取、AI 推理、安全分析——每个阶段是独立的模块，模块之间通过线程安全的数据包队列解耦。"
>
> "所有分析结果同时写入 SQLite 数据库，数据库采用三表设计——会话表、流表、分类记录表——并建有 8 个索引加速查询。"
>
> "最下层是操作系统网络栈，通过 Npcap 驱动从物理网卡捕获原始数据帧，支持以太网和 WiFi 802.11 两种链路层。"

**操作：**
- 打开架构图
- 鼠标从下往上指：网卡→后端管道→数据库→前端
- 每指到一个模块停顿 1-2 秒

### 2.4 实时抓包演示（2:30-5:30）⭐ 核心段落

**这是视频最重要的 3 分钟，需要放慢节奏，让观众清楚地看到每个组件的变化。**

**画面：** 浏览器仪表盘全屏

**操作 A：点击"开始抓包"按钮**

> "现在我选择 WiFi 网卡，点击开始抓包。系统启动 scapy 的 sniff 函数，通过 Npcap 驱动在物理网卡上捕获数据包。每个数据包经过六阶段管道处理后，分类结果通过 WebSocket 推送到前端，同时写入 SQLite 数据库。"

**操作 B：指向统计卡片（5 秒后）**

> "注意顶部四张统计卡片的变化——总流量从 0 开始增长，当前速率实时波动，总包数持续累加，运行时长计时器开始走动。每张卡片左侧有不同颜色的装饰条——青色代表流量、紫色代表速率、粉色代表包数、琥珀色代表时长。这些数据每 5 秒通过 statistics 事件推送一次。"

**操作 C：指向饼图（10 秒后，等有一些数据）**

> "左侧环形饼图展示 AI 分类的类别分布。可以看到网页流量占比最高，其次是视频和下载。每种颜色代表一个应用类别。鼠标悬停会弹出 tooltip，显示具体数值和百分比。这个图表每分钟随 statistics 事件自动更新。"

**操作 D：指向折线图（等 15-20 秒，积累一些数据点）**

> "右侧面积图按秒级粒度展示各类别的识别趋势。和传统的累积折线图不同，我们采用的是滑动窗口设计——只显示最近 60 秒的数据，每秒一个数据点。每条线下方有半透明的渐变色填充，视觉上更加柔和。注意这些线的波动——不同的时间段，不同类别的流量此起彼伏，真实反映了网络活动的变化。"

**操作 E：指向表格**

> "下方表格展示最近 20 条识别记录，更多历史记录会存放在我们之前的数据库中，可供查看。每条记录包含时间、源 IP、目标 IP 后面跟着蓝色的组织标签——可以看到这条流量发往了阿里 DNS、这条发往了 Google。协议列用彩色徽章区分 TCP 和 UDP。AI 分类列展示识别结果。最后一列是风险等级——大部分是绿色的'安全'，但如果出现黄色甚至红色标记，说明安全引擎检测到了异常。"

**操作 F：点击一行展开详情面板**

> "点击任意一行可以展开数据包的完整五层协议详情。面板分为两个部分：上面两张卡片是安全分析结果——左边是流量识别，告诉你这条流量从哪个组织发往哪个组织，具体是什么服务类型；右边是安全评估，显示风险等级和评分，如果有风险原因会逐条列出。下面四个折叠面板按五层模型展示——数据链路层的 MAC 地址、网络层的 IP 和 TTL、传输层的端口和 TCP 标志位、应用层的载荷大小和 AI 分类。"

**操作：**
- 每个操作之间停顿 3-5 秒
- 鼠标移到哪讲到哪，移动要慢
- 点击行展开详情时，上下滚动展示所有面板

### 2.5 历史回查（5:30-6:30）

**操作：** 先点击"停止抓包"，再点击"抓包记录"按钮

> "点击停止后，会话数据自动保存到数据库。现在打开抓包记录弹窗。左侧是所有历史会话的列表，每条显示网卡名称、时间、状态和数据量。顶部有日期筛选器，可以按日期范围过滤会话。每个会话右侧还有一个删除按钮。"
>
> "选择一个会话，右侧默认进入五层分析视图。这里展示的是整个会话所有数据包按五层模型聚合后的统计信息。"
>
> "物理层——展示包大小分布柱状图和按分钟聚合的速率时间线。"
>
> "数据链路层——展示所有出现过的源 MAC 和目的 MAC 地址，以及帧类型分布。"
>
> "网络层——展示 IP 地址的 Top 10 排名、TTL 值的分布范围，以及 IPv4/IPv6 的比例。"
>
> "传输层——展示 TCP/UDP 的占比、目的端口的 Top 10 热力图、以及 TCP 标志位（SYN/ACK/FIN 等）的分布情况。"
>
> "应用层——展示 AI 分类的分布直方图、每个类别的平均置信度、以及服务端口的识别结果（如 HTTPS 最多、DNS 次之）。"
>
> "切换到原始记录视图，可以看到这个会话下的所有分类记录。同样点击任意一行可以展开五层详情。"

**操作：**
- 点击一个历史会话
- 逐个点击五层标签（物理层→数据链路层→网络层→传输层→应用层）
- 每个标签停留 5 秒，鼠标指出一个关键数据点
- 切换到原始记录

### 2.6 代码亮点（6:30-8:00）

**画面：** 切换到 VS Code，依次打开三个文件

**要讲的内容（不是逐行读代码，而是讲设计思路）：**

> "最后我展示三个代码亮点，说明系统背后的核心技术。"
>
> **文件 1：`pipeline/packet_parser.py`**
>
> "这是五层协议解析器，83 行代码实现了从链路层到应用层的完整字段提取。它的核心设计是兼容性——旧版 scapy 中 IPv6 位于 inet6 模块、Dot11 802.11 层可能不存在、get_windows_if_list 函数仅在 2.5 以上版本可用。我们全部使用 try/except ImportError 做梯度降级。注意主函数 parse_packet 的返回 dict 包含 18 个字段，覆盖了 MAC 地址、IP、端口、TTL、TCP 标志位、窗口大小、载荷大小等关键信息。"
>
> **文件 2：`pipeline/inference.py`**
>
> "这是 AI 分类器的核心，采用三级降级策略。第一级加载训练好的 RandomForest 模型做 predict_proba 推理，100 棵树，最大深度 12，防止过拟合。如果模型文件不存在或加载失败，自动降级到第二级——七条启发式规则。例如 HTTPS 加高速率加大包判定为视频流、DNS 查询判定为网页、高频小包加短间隔判定为游戏。这些规则基于网络协议的实际行为特征设计。如果规则也匹配不了，最后用随机分类兜底。这个三级设计保证了系统在任何环境下——有模型、没模型、甚至是异常状态——都能正常运行。"
>
> **文件 3：`pipeline/security_analyzer.py`**
>
> "这是安全分析引擎，包含两个核心能力。第一是 IP 情报库，内置 40 多条规则，能够识别 Google、Cloudflare、Microsoft、腾讯、阿里、字节跳动等主流互联网服务商的 IP 范围。第二是 8 维度风险评估——从端口风险到横向移动检测。每个维度独立打分，最后累加成 0-100 的总评分，映射到四色等级。"
>
> "整个系统的模块化设计意味着——如果想换一个更好的分类模型，只需要修改 inference.py；想接入外部威胁情报 API，只需要修改 security_analyzer.py。其他模块完全不受影响。"

**操作：**
- 文件 1：滚动到主函数 `parse_packet()`，高亮返回 dict 的字段
- 文件 2：滚动到 `_heuristic_predict()` 函数，指出几条规则
- 文件 3：滚动到 `IP_INTEL` 列表和风险评估函数

### 2.7 总结（8:00-10:00）

**画面：** 回到仪表盘首页，或者展示一张总结 PPT

**要讲的内容：**

> "最后总结一下本项目的三个创新点。"
>
> "第一，五层模型与 AI 分类的深度结合。我们不仅做应用层分类，而是从物理层的包大小分布到应用层的 AI 分类，每一层都有独立的统计维度和可视化面板。这种纵深分析方式在同类系统中是比较少见的。"
>
> "第二，无外部依赖的安全分析能力。我们的 IP 情报库和 8 维风险评估引擎完全离线运行，不需要调用任何云 API，数据不出本地，这是网络流量分析场景下非常重要的隐私保护考量。"
>
> "第三，三级弹性降级机制。从真实抓包到模拟模式、从 ML 模型到启发式规则再到随机兜底，系统在任何运行环境下——有管理员权限还是没有、有 Npcap 驱动还是没有、有训练模型还是没有——都能正常工作。"
>
> "未来可以扩展的方向包括：集成 Snort/Suricata 入侵检测规则引擎实现对已知攻击的精确识别、接入 AbuseIPDB 和 AlienVault OTX 等外部威胁情报源提升安全分析准确度、引入 LSTM 或 Transformer 等深度学习方法进一步提升加密流量分类的准确率。"
>
> "以上就是我的项目介绍，谢谢各位老师，请提问。"

**操作：**
- 停在仪表盘首页
- 最后 5 秒冻结画面

## 三、录制技巧

### 3.1 语速与节奏

| 段落 | 语速 | 原因 |
|------|------|------|
| 开场 | 中速 | 建立第一印象 |
| 架构 | 中偏慢 | 信息量大，需要消化 |
| **实时演示** | **偏慢** | 核心段落，观众需要看清变化 |
| 历史回查 | 中速 | 辅助功能展示 |
| 代码亮点 | 中速 | 只讲逻辑不讲语法 |
| 总结 | 中速 | 收尾，加强记忆 |

### 3.2 鼠标操作

- 移动要**慢且平滑**，不要晃
- 讲到哪，鼠标指到哪
- 不要画圈、不要反复点击
- 推荐使用 OBS 的鼠标高亮插件

### 3.3 画面切换

| 场景 | 建议画面 |
|------|----------|
| 开场 | 仪表盘首页全屏 |
| 架构 | 实验报告文档 / 设计报告 |
| 启动 | 终端 + 浏览器 左右分屏 |
| 实时演示 | **浏览器全屏**（这是展示重点） |
| 历史回查 | 浏览器全屏 |
| 代码 | VS Code 全屏，字体调大（16-18px） |
| 总结 | 仪表盘首页全屏 |

### 3.4 常见错误避免

| 错误 | 正确做法 |
|------|----------|
| 语速过快 | 每句话之间停顿 1-2 秒 |
| 鼠标乱晃 | 手离鼠标，只在需要指的时候动 |
| 读代码 | 讲逻辑，不逐行读 |
| 提前没准备数据 | 提前跑 2-3 次抓包，确保有历史会话 |
| 后端报错 | 提前重启一次确保干净状态 |
| 浏览器有其他标签 | 关闭所有无关标签 |
| 录到桌面通知 | 开启免打扰模式 |

## 四、备用方案

### 4.1 如果抓包没有数据

- 切换到模拟模式（`config.py` 中设置 `CAPTURE_MODE = "simulated"`）
- 说明"这里演示的是模拟模式，逻辑和真实模式完全一致，只是数据来源不同"

### 4.2 如果前端卡顿

- 刷新浏览器（`Ctrl+Shift+R`）
- 检查后端终端是否有报错
- 提前 5 分钟启动，确保 WebSocket 连接稳定

### 4.3 如果时间超了

可快速跳过的段落（按优先级）：
1. 历史回查可压缩到 30 秒（只展示五层分析，不展示原始记录）
2. 代码亮点可压缩到 1 分钟（只展示 inference.py 三级降级）
3. 启动演示可压缩到 30 秒（后端已经在运行，只做简单说明）

可**绝对不能压缩**的段落：
- 实时抓包演示（核心展示）
- 总结（创新点陈述）

这个系统解决了传统流量分析工具的三大痛点。

第一，Wireshark 这类工具需要网络专家逐个数据包手动分析，学习成本非常高，不适合非专业人员使用。

第二，现在互联网流量中，HTTPS 加密占比已经超过百分之九十五。传统的深度包检测技术需要解密后才能分析载荷内容，在 TLS 加密面前基本失效。

第三，市面上缺少一个集成了实时分析、安全评估、历史回溯的一体化平台。做流量分析的可能没有安全评估，做安全监控的可能没有历史回查。

--

我们这个系统实现了从物理层抓包到应用层 AI 分类的全链路自动化分析。同时还加入了五层模型可视化、安全风险评估、GeoIP 物理地址识别等扩展功能。

一句话概括：它就是网络流量的"智能体检仪"——自动分析每一包数据是什么应用、有没有风险、从哪里来到哪里去。

### 【系统架构】 0:40-1:30

**（画面：切换到实验报告中的架构图）**

先看一下系统的整体架构。从上到下分为三层。

--

最上层是浏览器前端。使用 React 18 加 Vite 构建，通过两种协议与后端通信——HTTP REST 处理启停控制和历史查询请求，WebSocket 接收实时推送的分类结果和统计数据。

仪表盘包含七个组件：控制栏负责网卡选择和启停控制；四张统计卡片展示总流量、速率、包数和时长；两个 ECharts 图表——饼图展示类别分布，面积图展示实时趋势；下方的记录表格展示每条流量的详细信息；点击任意一行可以展开五层协议详情面板；还有一个历史记录弹窗用于回查过往会话。

--

中间层是 Flask 后端，这是系统的核心。它包含三个子系统。

REST API 模块暴露出七个接口端点，处理前端的各种请求。

WebSocket 模块基于 Socket.IO 协议，负责把分析结果实时推送到前端。推送频率是每两秒一条分类结果，每五秒一条统计汇总。

数据管道模块是整个系统的大脑，包含六个串行处理阶段。

第一个阶段，capture_manager 通过 scapy 加 Npcap 驱动从物理网卡捕获原始数据帧。

第二个阶段，packet_parser 把原始数据包解析成标准化的字典结构，提取出 MAC 地址、IP、端口、TTL、TCP 标志位、载荷大小等十几个关键字段。

第三个阶段，flow_reassembly 按五元组——源 IP、目的 IP、源端口、目的端口、协议——把离散的数据包聚合成网络流，并按照协议类型做超时清理。TCP 流六十秒无活动自动过期，UDP 三十秒，ICMP 十秒。

第四个阶段，feature_extractor 从每条流中提取三十维统计特征向量，包括包长分布、时间间隔、端口编码、协议标记等等。

第五个阶段，inference 推理分类器使用三级策略——先尝试机器学习模型，失败则用启发式规则，再失败用随机兜底——输出七个应用类别中的一个和对应的置信度。

第六个阶段是扩展阶段，包含三个分析模块。security_analyzer 做八维度安全风险评估并输出风险等级。session_analytics 对整个会话的数据做五层聚合统计。geoip 通过离线数据库查询 IP 的物理位置。

--

所有分析结果同时写入 SQLite 数据库。数据库采用三表设计——capture_session 会话表记录每次抓包的元信息；flow 流表按五元组记录每条流的状态；classification 分类记录表存储每一次推理的完整结果。三张表之间通过外键关联，并建立了八个索引加速查询。

--

最下层是操作系统网络栈，通过 Npcap 内核驱动从物理网卡捕获原始数据帧。系统同时支持以太网和 WiFi 802.11 两种链路层。

### 【代码亮点】 6:30-8:00

**（画面：切换到 VS Code，字体大小调到 16-18px）**

最后我展示三个核心代码文件，说明系统背后的关键技术。

--

**第一个文件，packet_parser.py，五层协议解析器。**

**（打开文件，滚动到 parse_packet 函数）**

这一段八十几行代码实现了从链路层到应用层的完整字段提取。它接收 scapy 的原始数据包对象，输出一个包含十八个字段的标准化字典。

注意这里的兼容性设计。IPv6 在旧版 scapy 中位于 inet6 模块，新版才合并到 inet；Dot11 这个 802.11 协议层在极旧版本的 scapy 中可能根本不存在；get_windows_if_list 这个函数仅在 2.5 以上版本可用。我们全部使用 try/except ImportError 做梯度降级——先尝试新版路径，失败了退回旧版路径，再失败就跳过该功能。这保证了系统在 scapy 2.4 到 2.6 的所有版本上都能正常运行。

**（滚动到 return 语句，高亮字段列表）**

返回的字典中包含 MAC 地址、IP、端口、TTL、TCP 标志位字符串、窗口大小、载荷大小等十八个字段——覆盖了 TCP/IP 五层模型的每一层。

--

**第二个文件，inference.py，AI 分类器的三级降级策略。**

**（打开文件，滚动到 predict 方法）**

这是分类器的核心入口，采用三级降级设计。

第一级，尝试加载训练好的 RandomForest 模型做推理。模型有 100 棵决策树，最大深度限制为 12 以防止过拟合。训练数据量较小的时候，深度太深会导致模型记住了训练集的具体模式而非学习一般规律。这一级的输出是 predict_proba 概率分布加上 argmax 取最高概率的类别。

**（滚动到 _heuristic_predict 方法）**

如果模型文件不存在或加载失败——比如第一次运行还没有训练模型——系统自动降级到第二级，七条启发式规则。

举个例子，这条规则：如果目的端口是 443 且是 TCP 协议、大包比例超过百分之三十、速率也较高，判定为视频流。这是基于一个事实观察——YouTube、B站等视频平台会通过 HTTPS 持续发送大量接近 MTU 的数据包。

再比如这条：如果小包比例超过百分之六十，而且包间隔很短，判定为在线游戏。因为游戏客户端和服务器之间需要高频交换小数据包来同步状态。

**（滚动到最下面）**

如果连规则也匹配不了——比如遇到了完全未知的流量模式——最后用随机分类兜底。虽然准确率低，但保证了系统在任何情况下都不会崩溃。

这个三级设计是平台高可用性的核心。

--

**第三个文件，security_analyzer.py，AI 安全分析引擎。**

**（打开文件，滚动到 IP_INTEL 列表）**

这个引擎包含两个核心能力。

第一是 IP 情报库。这里列出了四十多条规则，覆盖了 Google、Cloudflare、Microsoft、Amazon、腾讯、阿里、字节跳动、百度等主流互联网服务商的 IP 范围。每条规则包含 IP 网段、组织名称和服务类型。查询时通过 Python 标准库的 ipaddress 模块做 CIDR 匹配，时间复杂度 O(n)，对于四十条规则来说完全够用。

**（滚动到 analyze_traffic 函数）**

第二是八维度风险评估。从端口风险开始——检查目标端口是否在已知高危端口列表中，如 445 是 SMB 永恒之蓝漏洞端口、3389 是远程桌面爆破目标、4444 是 Metasploit 渗透框架的默认端口——到内网横向移动检测——如果一个内网 IP 向另一个内网 IP 发起了 SMB 或 RDP 连接，可能意味着攻击者已经在内部网络中横向移动。

每个维度独立打分，最后累加成零到一百的总评分。评分超过 50 标记为高危，25 到 49 为可疑，10 到 24 需要注意，低于 10 为安全。

--

整个系统的模块化设计意味着——如果想换一个更好的分类模型，只需要修改 inference.py；想接入外部的威胁情报 API，只需要修改 security_analyzer.py；其余模块完全不受影响。每个模块有明确的输入输出契约，可以独立测试、独立替换。

### 【总结】 8:00-10:00

**（画面：回到仪表盘首页）**

最后总结一下本项目的三个创新点。

--

第一，五层模型与 AI 分类的深度结合。

我们不仅做应用层的流量分类。而是从物理层的包大小分布，到数据链路层的 MAC 地址和帧类型，到网络层的 IP 和 TTL，到传输层的 TCP 标志位和端口，再到应用层的 AI 分类和载荷分析——每一层都有独立的统计维度和可视化面板。这种纵深分析方式在同类开源系统中是比较少见的。它不只是告诉用户"这是什么应用"，还告诉你"这个数据包在网络协议的每一层上发生了什么"。

--

第二，无外部依赖的安全分析能力。

我们的 IP 情报库、八维度风险评估引擎、GeoIP 物理定位——全部离线运行，不需要调用任何云 API。这意味着所有流量数据都在本地处理，数据不出用户的电脑。在网络流量分析这个场景下，隐私保护是一个非常重要的考量——你肯定不希望把全家的上网记录上传到某个第三方服务去做分析。

--

第三，三级弹性降级机制。

系统在三个层面上都有降级保障。抓包层面——有管理员权限就用 Npcap 真实抓包，没有权限自动切换到模拟数据生成器。分类层面——有训练模型就用机器学习推理，没有模型就用启发式规则，规则也匹配不了就随机兜底。环境检测层面——兼容 scapy 2.4 到 2.6 的所有版本，自动适配不同版本间的 API 差异。

这种层层递进的弹性设计，保证了系统在各种运行环境下都能正常工作。

--

未来可以扩展的方向。

在安全检测方面，可以集成 Snort 或 Suricata 的入侵检测规则引擎，实现已知攻击的精确识别。

在威胁情报方面，可以接入 AbuseIPDB 和 AlienVault OTX 等外部威胁情报源，进一步提升安全分析的准确度。

在 AI 模型方面，可以引入 LSTM 或 Transformer 等深度学习方法，进一步提升加密流量分类的准确率——目前学术界已经有论文报告 Transformer 在加密流量分类上可以达到百分之九十五以上的准确率。

在部署形态方面，可以将系统容器化，通过 Docker 一键部署到服务器上，实现多客户端远程监控。

## 附录二：逐字稿使用说明

1. **--** 表示停顿 1-2 秒，让观众消化信息
2. **加粗的文字** 是操作动作，不需要念出来
3. 方括号 `[...]` 内的内容为可选补充，如果时间紧可以跳过
4. 建议录制前通读两遍，熟悉整体节奏
5. 不需要逐字背诵——理解大意后用自己的话讲出来效果更好
6. 如果某一段讲得不好，停下来重讲这一段即可，后期可以剪辑
